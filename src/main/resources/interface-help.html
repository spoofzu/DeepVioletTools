DeepViolet Interface Help

This guide explains how to use the DeepViolet interface and how to read and interpret scan reports. Each section of a scan report is documented below with explanations of the terminology, values, and warning messages you may encounter.

1. How to Read a Report

A DeepViolet scan report is organized into sections, each enclosed in [Section Name] brackets. Sections appear in the order they are enabled in System > Settings > Engine.

Color coding: The results pane uses syntax highlighting to distinguish different types of information:

  • Section headings appear in the heading color
  • Subsection labels appear bold in the subsection color
  • Key=Value pairs show keys and values in distinct colors
  • Warnings are highlighted in the warning color
  • Risk deductions are color-coded by severity: CRITICAL, HIGH, MEDIUM, LOW, and INFO each appear in a distinct color

Warning markers: Issues requiring attention are wrapped in >>>...<<< markers. For example, >>>MISSING<<< indicates a missing security header, and >>>UNTRUSTED<<< flags an untrusted certificate.

2. Saving and Exporting Reports

Click Save to export the current scan results. A file chooser dialog opens with a format selector panel on the right side. Supported formats:

FormatExtensionDescription
ASCII Text.txtPlain text report

The last-used format and folder are remembered between sessions.

Scan results are also auto-saved as text files to ~/DeepVioletTools/ui/scans/ each time a scan completes. These auto-saved files are available for review in the AI Assistant tab.

Heat Map Results

After a scan completes, results are displayed in four heat map tabs:

  • Risk — overall TLS risk assessment scores across hosts
  • Ciphers — cipher suite support across hosts
  • Connection — connection properties across hosts
  • Fingerprint — TLS fingerprint components across hosts

Cell colors indicate pass (green), fail (red), or mixed results. The Host Details tab shows the full individual report for any host in the scan.

Status Bar

During a scan, the status bar at the bottom shows the current worker activity, for example: worker 1: TLS risk assessment. When multiple workers run in parallel, they are separated by pipes: worker 1: TLS risk assessment | worker 2: Security headers.

Saving Results

Scan results are auto-saved to ~/DeepVioletTools/ui/scans/ when a scan completes. You can also click Save to export in Text, HTML, PDF, or JSON format.

4. Delta Scanning

Delta scanning compares two previously saved .dvscan files to identify changes in a server's TLS configuration over time. Click the Delta toolbar button to open the comparison dialog.

How to Use

  1. Select a Base scan file (the earlier snapshot)
  2. Select a Target scan file (the later snapshot)
  3. Click Compare to generate the delta report

Delta Report

The delta report shows per-host changes organized by category:

  • Risk changes — score and grade differences
  • Cipher changes — added or removed cipher suites
  • Fingerprint changes — TLS fingerprint differences
  • Added hosts — hosts present in the target but not the base
  • Removed hosts — hosts present in the base but not the target

The status bar summarizes the delta: changed, added, and removed host counts.

CLI: Use --delta base.dvscan,target.dvscan to compare two scan files from the command line.

5. AI Assistant

The AI Assistant tab provides a terminal-style chat interface for asking questions about scan results. To use it:

  1. Enable AI in System > Settings > AI > Configuration and configure a provider
  2. Switch to the AI Assistant tab
  3. Choose a Scan Type using the radio buttons: Individual for single-host scans or Multi-Target for multi-target scans
  4. Select a scan from the scan selector dropdown (shows saved scans of the selected type)
  5. Type a question in the text field and click Send (or press Enter)

The AI receives the selected scan report as context and can answer questions about the server's TLS configuration, certificate details, security headers, vulnerabilities, and remediation steps. Chat history is maintained within each scan context so follow-up questions understand previous context. Selecting a different scan from the dropdown resets the chat history.

Supported Providers

ProviderAPI KeyNotes
AnthropicRequiredCloud-hosted Claude models
OpenAIRequiredCloud-hosted GPT models
OllamaNot requiredLocal models; configure the endpoint URL in Settings

API keys are encrypted with AES-GCM before being stored locally. Chat history is logged to ~/DeepVioletTools/ui/aichat.log.

Terminal Colors

The AI terminal appearance can be customized in System > Settings > AI > Terminal Colors. Nine color controls are available for background, system text, user prefix/text, AI prefix/text, error text, and selection colors.

6. Card Layout Editor

The Cards sub-tab in System > Settings > Reporting lets you customize the appearance of host cards used in scan results.

Grid Editor

The card layout editor displays a configurable grid. Metadata elements from the palette on the left can be placed into grid cells. Elements appear as colored tiles showing their name and a small alignment dot.

7. Custom Overlays

DeepViolet supports two custom overlay features that let you extend or replace built-in data with your own YAML definitions. Both are configured in System > Settings.

Custom Cipher Map

The Cipher Map settings tab lets you replace the built-in cipher suite map with a custom YAML definition. When enabled, cipher suite strength ratings in scan reports reflect your custom map instead of the defaults.

  • Enable — check "Enable custom cipher map" to activate
  • Edit — paste or type YAML directly in the text area
  • Load File — import a .yaml or .yml file
  • Save to File — export the current YAML to a file
  • Clear — remove the custom map and revert to defaults

The YAML format uses a cipher_suites array with id, names (IANA, OpenSSL, etc.), strength (STRONG, MEDIUM, WEAK, CLEAR), and tls_versions fields. See the placeholder text in the settings tab for an example.

CLI: Use --ciphermap <path> to load a custom cipher map from a YAML file. The CLI flag overrides the saved GUI preference.

User Risk Rules

The User Risks settings tab lets you define custom risk assessment rules that are merged with the built-in system rules during scoring. Custom rules use USR- prefixed IDs and appear alongside system findings in the risk assessment.

  • Enable — check "Enable user risk rules" to activate
  • Edit — paste or type YAML directly in the text area
  • Load File — import a .yaml or .yml file
  • Save to File — export the current YAML to a file
  • Clear — remove user rules (only system rules will apply)

The YAML format uses a categories map with rule definitions containing id, description, score, when, and enabled fields. See the placeholder text in the settings tab for an example.

CLI: Use --riskrules <path> to load user risk rules from a YAML file. The CLI flag overrides the saved GUI preference.

8. Report Header

The header section provides context about the scan environment:

  • DeepViolet version and report timestamp
  • Java version — the JVM running the scan, which determines available TLS capabilities
  • Trust store — the file path of the Java trust store (cacerts) used to verify certificate chains
  • Log file — location of the detailed scan log for troubleshooting

9. TLS Risk Assessment

The risk assessment provides an overall security evaluation of the target server.

Scoring

The Overall Score is calculated out of 115 points. Points are deducted for security weaknesses found during the scan. The score maps to a letter grade:

GradeScore RangeRisk Level
A+100%MINIMAL
A90–99%LOW
B80–89%MODERATE
C70–79%ELEVATED
D60–69%HIGH
F<60%CRITICAL

Category Graph

The bar chart shows per-category scores using colored blocks:

  • Pass blocks (green) — points earned
  • Inconclusive blocks (amber) — could not be determined
  • Fail blocks (red) — points deducted

The scale (blocks per bar) is configurable in System > Settings > Reporting > Report Colors > TLS Risk Assessment > Scale.

Deductions

Below each category, individual deductions are listed in the format RULE-ID [SEVERITY] Description (score: N.NN) and color-coded by severity:

  • CRITICAL — severe security issue
  • HIGH — significant vulnerability
  • MEDIUM — notable weakness
  • LOW — minor issue, small point deduction
  • INFO — informational observation, minimal or no point deduction

Severity colors are configurable in System > Settings > Reporting > Report Colors > TLS Risk Assessment.

Other & SAN Disclosure

The Other category (20 points) checks for miscellaneous issues including TLS compression, client authentication, fingerprint availability, and SAN (Subject Alternative Name) information disclosure. Certificates covering many SANs reveal an organization's hosted services, staging servers, and internal naming conventions, increasing the visible attack surface.

10. AI Evaluation

When AI is enabled (System > Settings > AI), the report can include an AI-powered analysis section after the risk assessment. The scan report generated so far is sent to the configured AI provider, which returns a structured analysis.

The AI evaluation typically includes:

  • Executive Summary — overall TLS security posture assessment
  • Per-Finding Analysis — for each risk assessment finding: what it is, why it matters, and remediation. Each finding header is color-coded by severity (CRITICAL, HIGH, MEDIUM, LOW, INFO) using the same colors as the TLS Risk Assessment deductions.
  • Positive Findings — security measures that are properly configured
  • Recommendations — prioritized action items

The system prompt that controls the analysis can be customized in System > Settings > AI > Prompts > Report System Prompt.

If the AI provider is unreachable or returns an error, the section shows a warning message with the failure reason.

11. Host Information

This section shows DNS resolution details for the target:

  • Hostname — the name you entered
  • IP address — the resolved IPv4 or IPv6 address
  • Canonical name — the canonical hostname returned by DNS (may differ if the host is a CNAME alias)

12. HTTP Response Headers

Raw HTTP response headers returned by the server. These headers control browser behavior, caching, content negotiation, and more. They are listed as key=value pairs exactly as received from the server.

This section provides the raw data; for security-specific analysis, see the Security Headers Analysis section below.

Checks for the presence and values of security-critical HTTP response headers. Headers marked >>>MISSING<<< are not present and should be configured.

14. Connection Characteristics

TLS session properties and socket configuration details for the connection, including protocol versions, keepalive settings, buffer sizes, and TCP nodelay. These reflect how the underlying Java TLS socket is configured for the scan.

15. Cipher Suites

A cipher suite defines the set of cryptographic algorithms used for a TLS connection: key exchange, authentication, encryption, and message authentication. This section lists all cipher suites the server is willing to negotiate.

Strength Ratings

RatingMeaning
STRONGModern, secure cipher (e.g., AES-256-GCM, ChaCha20-Poly1305)
MEDIUMAcceptable but not ideal (e.g., AES-128)
WEAKShould be disabled (e.g., 3DES, RC4)
CLEARNo encryption (plaintext) — never acceptable

Naming Conventions

Cipher suite names vary by library. DeepViolet supports four naming conventions, configurable in System > Settings > Engine > Cipher Suite Naming:

  • IANA — standard names (default, e.g., TLS_AES_256_GCM_SHA384)
  • OpenSSL — OpenSSL names (e.g., AES256-GCM-SHA384)
  • GnuTLS — GnuTLS names
  • NSS — Mozilla NSS names

Servers supporting only WEAK or CLEAR ciphers are at significant risk. Best practice: enable only STRONG ciphers and disable all others.

16. Certificate Chain

TLS servers present a certificate chain for identity verification. The chain typically includes:

  1. End-Entity certificate — the server's own certificate
  2. Intermediate CA(s) — certificates linking the end-entity to a trusted root
  3. Root CA — a trusted certificate authority in the Java trust store

Trust States

StateMeaning
TRUSTEDChain verified against the Java trust store
>>>UNTRUSTED<<<Chain could not be verified (self-signed, unknown CA, or missing intermediates)
>>>UNKNOWN<<<Trust state could not be determined

Validity States

StateMeaning
VALIDCertificate is within its validity period
>>>EXPIRED<<<Certificate validity period has ended
>>>NOT YET VALID<<<Certificate's Not Before date is in the future

Expiration Warnings

Certificates expiring within 30 days generate a warning. Expired certificates show the number of days since expiration.

Key Algorithms and Sizes

The report shows the public key algorithm (RSA, EC), key size in bits, and for ECC certificates, the named curve (e.g., secp256r1). Larger keys generally provide stronger security but slower handshakes.

17. Revocation Status

Certificate revocation checking verifies that certificates have not been invalidated by their issuing CA. DeepViolet checks multiple revocation mechanisms:

OCSP (Online Certificate Status Protocol)

Real-time revocation check against the CA's OCSP responder. The report shows the responder URL, status (GOOD, REVOKED, UNKNOWN), response time, and signature validation. "Unauthorized" errors are common for intermediate/root CA certificates and are usually normal.

OCSP Stapling

The server pre-fetches its OCSP response and includes it in the TLS handshake. This is faster and more privacy-friendly than clients querying the OCSP responder directly. Look for PRESENT (good) or NOT PRESENT (consider enabling on the server).

Must-Staple

An extension (RFC 7633) that tells browsers to require OCSP stapling. If a certificate has Must-Staple set but the server does not provide a stapled response, browsers will reject the connection. A warning is shown when this mismatch is detected.

CRL (Certificate Revocation List)

A list of revoked certificate serial numbers published by the CA. The report shows the CRL distribution point URL, download time, and CRL size.

OneCRL

Mozilla's centralized revocation list for high-priority certificate revocations. A certificate found in OneCRL has been explicitly blocked by Mozilla.

Certificate Transparency (SCTs)

Certificate Transparency is a system of public logs that record all issued certificates. Signed Certificate Timestamps (SCTs) prove a certificate was submitted to a CT log.

The report shows:

  • Embedded SCT count — SCTs embedded in the certificate itself
  • TLS Extension SCTs — SCTs delivered via TLS extension
  • OCSP Staple SCTs — SCTs delivered via stapled OCSP response
  • CT log identity — operator, URL, and log state (usable, retired, rejected)
  • SCT signature verification — VALID, INVALID, or UNKNOWN_LOG

An >>>INVALID<<< signature is a serious finding that may indicate a forged or corrupted SCT.

18. TLS Server Fingerprint

The fingerprint characterizes a server's TLS implementation by sending 10 different probe handshakes and hashing the combined results.

Probes

The 10 probes test different TLS configurations: standard and reversed cipher order, ALPN negotiation, non-ECC suites, different protocol versions (TLS 1.1, 1.2, 1.3), and forward secrecy support. Each probe result is either OK (handshake succeeded) or FAIL (did not complete).

Use Cases

  • Server grouping — servers with the same fingerprint likely run the same TLS stack
  • CDN detection — identify servers behind the same CDN or load balancer
  • Change tracking — detect when a server's TLS configuration changes over time

The fingerprint is displayed as colon-delimited hex octets. An Extension Hash provides additional characterization based on TLS extensions offered by the server.
